# 附件第一阶段设计

本文档用于收敛 **“笔记级附件闭环”** 的第一阶段目标，不讨论全局附件库。

## 1. 目标

第一阶段只解决一篇笔记内的附件可见、可跳转、可区分状态、可安全管理。

本阶段暂不解决：
- 全局附件检索
- 跨笔记复用追踪
- 重复附件去重
- 附件批量治理

## 2. 用户问题

当前附件能力存在 4 个核心断点：

1. 粘贴/上传图片后，正文里有图，但用户不确定附件是否真正落盘。
2. 附件列表能看见文件名，但很难反向定位到正文中的引用位置。
3. 删除正文图片后，附件是否还保留、是否还能恢复，没有明确规则。
4. 附件区操作入口太弱，缺少打开、复制链接、删除等管理动作。

## 3. 第一阶段产品规则

### 3.1 附件归属

- 附件默认归属于**当前笔记**。
- 第一阶段不做“独立于笔记存在”的附件对象管理台。
- 附件区仅展示当前笔记下的附件。

### 3.2 引用状态

每个附件在当前笔记内有两种状态：

- `已引用`：正文中已经出现该附件的内容链接。
- `未引用`：附件已上传并存储，但当前正文还没有引用。

第一阶段允许通过 **Markdown/正文中的内容 URL** 轻量推断引用状态。
后续正式进入附件闭环实现时，再升级为 `attachmentId` 强关联。

### 3.3 点击行为

- 点击 `已引用` 附件：直接跳转到正文中的对应位置。
- 点击 `未引用` 附件：不给危险操作，提示“尚未插入正文”。
- 附件行保留少量显式按钮，避免把所有动作堆到列表里。

### 3.4 右键菜单

附件区右键菜单作为主要管理入口，第一阶段建议放：

- 跳转到正文
- 打开附件
- 复制附件链接

后续进入完整闭环后，再补：

- 从当前笔记移除
- 彻底删除附件
- 重新插入到当前光标位置

### 3.5 删除策略

第一阶段不采用“删正文图片 = 立即删附件文件”的强绑定。

推荐两段式：

1. 删除正文图片后，附件先变成 `未引用`
2. 用户再在附件区决定：
   - 继续保留
   - 从当前笔记移除
   - 彻底删除

这样更安全，也更符合迁移资料时的恢复诉求。

## 4. 第一阶段数据规则

### 4.1 当前可行的轻量规则

当前可先沿用已有附件数据结构：

```json
{
  "id": "attachment-xxx",
  "noteId": "note-xxx",
  "fileName": "image.png",
  "mimeType": "image/png",
  "size": 12345,
  "storagePath": "storage/uploads/attachment-xxx-image.png",
  "createdAt": "2026-07-07T12:00:00.000Z"
}
```

在前端渲染层补两类派生字段：

- `contentUrl`
- `isReferenced`

这两个字段不要求立即落库存，可先作为显示层派生值使用。

### 4.2 第二步要升级的数据能力

当真正开始做“删除同步、引用恢复、正文反查附件”时，建议给图片块补：

- `attachmentId`

这样才能从“字符串匹配 URL”升级到“稳定的实体关联”。

## 5. 第一阶段验收标准

满足以下几点即可视为第一阶段准备完成：

1. 粘贴/上传图片后，附件区立即出现对应附件。
2. 附件区能明确显示 `已引用 / 未引用`。
3. 点击 `已引用` 附件可以跳到正文位置。
4. 附件右键菜单可完成“跳转 / 打开 / 复制链接”。
5. 删除正文图片时，不会静默丢失附件文件。

## 6. 第二阶段边界

只有当第一阶段闭环跑顺后，才建议开启第二阶段：

- 全局附件库
- 孤儿附件治理
- 跨笔记引用追踪
- 附件批量清理
- 重复附件识别

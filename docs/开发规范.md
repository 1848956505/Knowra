# AGENTS.md - AI Coding Agent 项目规范

> **致 AI Agent**：本文件是你的最高行为准则。在生成、修改任何代码前，必须严格遵循以下约束。不要试图为了“图方便”而违反这些规则。

## ⛔ 核心铁律 (Core Iron Rules)
1. **物理分层**：视图层(Presentation) → 业务层(Business) → 数据层(Data) → 基础设施(Infra)。依赖必须单向，严禁反向或跨层调用。
2. **单一职责与行数限制**：
普通业务源文件理想不超过 **250 行**；
超过 250 行必须自查是否职责过多，若发现职责混杂、函数过长、组件过大、层级边界不清或依赖过重，应主动拆分；
超过 400～500 行，若同时存在职责混杂或维护困难，应优先列入拆分队列；特殊文件允许例外并说明原因。
若职责过多，需要主动拆分并提供调用关系，严禁生成“上帝文件”。
允许例外：类型定义聚合文件、路由配置、迁移文件、自动生成文件、测试快照、Mock 数据、Schema 定义、图标/常量映射表。但例外文件不得混入业务逻辑。
注意：拆的是“职责边界”，不是“为了凑行数切片”。
3. **零硬编码**：严禁在代码中硬编码颜色值、间距、API URL、密钥或魔法数字。必须使用设计令牌(Design Tokens)或环境变量/配置中心。
4. **多平台开发**：当前项目在Windows、MacOS、Ubuntu22.04这三个平台开发，你需要确保多平台可共用一套代码

## 🎨 前端规范 (Frontend Rules)
**核心目标**：确保极高的可维护性、UI 重构的便捷性、主题的一键更换。

### 1. 样式与逻辑绝对抽离 (Theme & Style)
- **禁止硬编码**：严禁在组件内写死颜色（如 `#fff`）、字号、阴影或间距。所有视觉值必须引用全局 CSS 变量或 Design Tokens（如 `var(--color-primary)`）。
- **样式集中管理**：主题变量必须集中在独立的样式文件中。确保未来更换主题时，只需修改 Token 文件，无需触碰业务组件。
- **状态与视图解耦**：交互状态（展开/折叠、选中）必须与业务数据状态分离，禁止在 DOM 节点上直接挂载业务逻辑。

### 2. 组件库与高复用 (Component Reusability)
- **统一组件库**：所有基础 UI 元素（按钮、标签、弹窗、列表项）必须沉淀到全局或模块级组件库中。
- **禁止重复造轮子**：生成新 UI 前，必须先检索现有组件库。若已有组件能适配，**必须复用**；仅在现有组件完全无法满足需求时，才允许开发新组件。
- **命名表达职责**：组件命名必须表达其业务职责（如 `SearchPanel`），业务组件应优先按业务职责命名，避免仅用 Left/Right/Top 等位置命名。纯布局组件可以使用 AppShell、NavigationRail、SplitPanel、EditorLayout 等布局语义命名，以利于后续 UI 重构。

### 3. 文件拆分与组件化 (File Splitting)
- **拒绝大杂烩**：严禁将视图、业务逻辑、API 请求挤在同一个文件中。必须按职责拆分为 `View`, `Hook/Store`, `Service` 等独立文件。
- **逻辑复用**：通用交互行为（如关闭弹窗、复制、下载）必须沉淀为可复用的 Hooks 或 Composables。

### 4. 布局与交互铁律 (UI/UX Principles)
- **外壳稳定与独立滚动**：应用外壳（导航/侧栏）必须常驻。侧边栏、主内容区、辅助面板必须**各自拥有独立的滚动容器**，严禁将滚动权交给浏览器全局 `window`。
- **禁止原生弹窗**：严禁使用浏览器原生的 `alert`, `confirm`, `prompt` 和原生文件对话框，必须使用项目内自定义的统一组件。
- **克制设计**：默认呈现核心数据，不堆砌装饰性元素。谨慎使用全局提示条和动效，避免打断用户阅读节奏。

## ⚙️ 后端规范 (Backend Rules)
> **致 AI Agent**：你懂“高内聚低耦合”，但你经常在以下 7 个后端工程细节上犯错。请重点规避：

### 1. 事务边界污染 (Long Transaction Anti-pattern)
- **禁止在事务中做 IO**：严禁在 `@Transactional`（或数据库事务块）内部调用第三方 API、发送 MQ 消息、发送邮件或操作 Redis。
- **正确做法**：事务只保护本地数据库操作。外部 IO 操作必须在事务提交后（如使用事件发布/监听机制）异步执行，防止长事务锁死连接。

### 2. 异常生吞与掩盖 (Silent Exception Catching)
- **禁止生吞异常**：严禁写 `catch (Exception e) {}` 或仅仅 `log.error` 后返回 `null`/默认值。这会导致上层业务无法感知错误，排查灾难。
- **正确做法**：必须抛出明确的业务异常（Business Exception）或系统异常，由全局异常处理器（Global Exception Handler）统一捕获并转换为标准响应格式（如 `{code, msg, data}`）。

### 3. 循环查库与 N+1 问题 (N+1 Query Problem)
- **禁止在循环中查库**：严禁在 `for`/`while` 循环内部执行数据库查询或外部 API 调用。
- **正确做法**：必须在循环外提取所有 ID，使用批量查询（如 SQL 的 `IN` 语句）或 ORM 的预加载（Eager Loading / `JOIN FETCH`）一次性获取数据，然后在内存中进行映射。

### 4. 实体泄漏 (Entity Leakage)
- **禁止直接返回 DB 实体**：严禁将数据库持久化对象（PO/Entity）直接作为 API 响应返回给前端。这会导致数据库结构变更直接破坏前端，甚至泄漏密码哈希等敏感字段。
- **正确做法**：必须在业务层或接口层将 PO 转换为 DTO（数据传输对象）或 VO（视图对象）。

### 5. 上帝方法 (God Method)
- **禁止巨型函数**：严禁将几十行甚至上百行的逻辑塞进一个 Service 方法中。
- **正确做法**：复杂业务必须拆分为明确的步骤函数，如 `validateInput()` -> `calculateBusiness()` -> `saveData()` -> `publishEvent()`，保持主流程函数的高可读性。

### 6. 并发与幂等性盲区 (Concurrency & Idempotency)
- **核心链路防护**：在涉及资金扣减、库存扣减、状态流转等核心业务时，禁止仅依赖数据库默认隔离级别。
- **正确做法**：必须显式考虑并发安全，使用乐观锁（Version 字段）、分布式锁或数据库唯一索引来保证幂等性，防止超卖或重复执行。

### 7. 接口层瘦身 (Thin Controller)
- **Controller 职责限制**：接口层/控制器只做三件事：请求参数绑定与校验、调用业务层、格式化统一响应。
- **禁止**：严禁在 Controller 中编写任何业务判断、数据转换或数据库操作代码。

### 2. 版本号体系

本项目使用 [SemVer 2.0.0](https://semver.org/lang/zh-CN/)：`MAJOR.MINOR.PATCH`。

- **当前稳定版本**：`2.0.6`（2026-07-05 起的官方起点）
- 之前的 `0.1.0` 是 npm 默认值从未更新过；`v1.x` 计划见 [`docs/已归档/`](已归档/)；`v2.0.0 ~ v2.0.5` 是 [`docs/V2.0开发计划.md`](V2.0开发计划.md) 的开发路线图
- **3 个 `package.json` 是版本号唯一来源**（`package.json` 根 + `apps/api/package.json` + `apps/web/package.json`），三者必须保持一致

### 3. 升版规则

| 变更类型 | 升幅 | 例子 |
| --- | --- | --- |
| bug 修复、构建遗漏、文档/脚本修复、依赖 patch 升级 | **PATCH** | `2.0.6` → `2.0.7` |
| 新功能、新模块、新增 HTTP 接口、不破坏现有接口 | **MINOR** | `2.0.6` → `2.1.0` |
| API 重命名、目录结构大规模迁移、数据库 schema 不兼容变更 | **MAJOR** | `2.0.6` → `3.0.0` |

- **不要自作主张升 MAJOR**。哪怕看起来破坏性大，也要先告诉人类再决定
- **MINOR 内部允许小步快跑**（攒波发版），每个 PATCH 单独发版成本太高
- **升版流程**：① 手动改 3 个 `package.json` ② 在 `docs/工程变更日志.md` 写新版本段 ③ 告诉人类 "git tag vX.Y.Z"（**Agent 不自己打 tag**）

### 4. Commit Message 规范

使用 [Conventional Commits](https://www.conventionalcommits.org/zh-hans/) 格式：

```
<type>(<scope>): <中文简述>

<中文详细说明（可选）>

<footer, 如 BREAKING CHANGE: ...>
```

**type**：`feat` `fix` `docs` `style` `refactor` `perf` `test` `chore` `ci` `revert`

**scope 建议**：`api` `web` `editor` `nav` `sidebar` `knowledge` `docs` `infra` `deploy` `pm2`

**BREAKING CHANGE 标记**：
- 标题行加 `!`：`feat(api)!: 重命名 /api/knowledge/spaces 为 /api/spaces`
- 或 footer 加：`BREAKING CHANGE: ...`
- 出现 BREAKING CHANGE 必须升 MAJOR，并明确告诉人类

**历史 commit 不重写**。`43cae67 feat: 初始化 Knowra 项目...` 已兼容新规范。

### 5. 多平台开发注意

本项目开发涉及 **macOS**（用户本地开发）/** Windows **（用户本地开发）/** Linux/Ubuntu **（阿里云 ECS 服务器）/** Web **（浏览器）四个端。

**跨平台脚本**：
- Shebang：`#!/usr/bin/env bash`
- 严格模式：`set -euo pipefail`
- 命令探测：`command -v pm2 >/dev/null 2>&1`（不要 `which`）
- **避免 `sed -i`**：macOS BSD `sed` 与 Linux GNU `sed` 行为不同。如需替换用 `perl -pi -e` 或写入临时文件
- **避免 `echo -e`**：用 `printf '%s\n' '...' >> file`
- 路径：用 Node.js `path.join` / `path.resolve`（不要字符串拼接 `/` 或 `\\`）
- 换行符：仓库统一 LF，必要时加 `.gitattributes`

**端口管理**：用 `scripts/dev-runtime-ports.js` 的 `readRuntimePorts` / `writeRuntimePort` / `resolveApiPort`，**不要硬编码** `3000` / `3001`。跨平台探测：`net.createServer().listen(port, '127.0.0.1')`

**Node.js / npm**：`>= 24.0.0` / `>= 11`（已在根 `package.json` 的 `engines` 字段约束）。服务端部署 Linux/Ubuntu，客户端开发 macOS / Windows + Web 验证

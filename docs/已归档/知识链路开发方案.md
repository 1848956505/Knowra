# 知识链路开发方案（Codex 执行版）

> 文档定位：本文件用于指导 Codex 在现有项目中逐阶段开发“知识库底座 → 知识提炼与考点体系 → RAG 知识库”整条知识链路。
>
> 本文件是实施方案，不是一次性代码生成清单。Codex 必须按阶段执行、逐步验证，不得将全部功能塞入一次改动。
>
> 建议存放路径：`docs/知识链路开发方案.md`

---

## 1. 项目目标

本项目的“知识”能力不只是保存笔记，也不只是提供 AI 问答，而是要将原始学习资料逐步转化为可训练、可评测、可追踪的学习结构。

完整链路如下：

```text
笔记原文
→ 重要内容标注
→ 知识内容
→ 考点
→ 试卷蓝图
→ 题目与试卷
→ 作答与批改
→ 掌握度
```

RAG 作为检索增强层，贯穿原文、知识内容和考点：

```text
笔记原文 / 知识内容 / 考点
→ 结构化切片与索引
→ 关键词检索 + 向量检索 + 结构化过滤
→ AI 问答、查重、关联、跨笔记出题与复习推荐
```

本方案聚焦“知识”链路本身，试卷、答题、批改和掌握度只定义接口边界，不在本阶段一次性全部实现。

## 2. 核心设计结论

### 2.1 不再把正文选区直接等同于知识点

正文中被选中的内容通常宽泛、上下文依赖较强，适合表达“这里重要”，但不一定是一个边界清晰、可直接出题的知识单元。

因此必须区分：

```text
标注：哪里重要
知识内容：这里讲了什么
考点：学习者需要会什么
```

### 2.2 知识内容与考点必须分离

- **知识内容（Knowledge Item）**：对知识本身的规范化表达。
- **考点（Assessment Point）**：围绕知识内容定义的可评测学习目标。

示例：

```text
知识内容：
Prim 算法每次选择一条连接已访问集合和未访问集合的最小权边，
逐步构造最小生成树。

考点：
1. 能够说明 Prim 算法解决的问题。
2. 能够解释 Prim 每轮的选边规则。
3. 能够手算 Prim 的执行过程。
4. 能够比较 Prim 与 Dijkstra 的贪心目标。
```

### 2.3 RAG 不替代结构化知识模型

RAG 负责“找到相关材料”，知识内容与考点负责“组织知识并定义学习目标”。

```text
RAG = 找材料
知识内容 = 组织材料
考点 = 定义评测目标
试卷蓝图 = 规划考试
题目 = 执行评测
```

### 2.4 AI 结果必须先进入候选态

AI 自动生成的知识内容、考点、关联关系和题目不得直接成为正式数据。

统一流程：

```text
AI 生成
→ 结构校验
→ 候选数据
→ 用户确认或修改
→ 正式数据
```

### 2.5 同时设计，分阶段开发

开发顺序为：

```text
阶段 0：现状审计与模型冻结
阶段 1：知识库底座与重要内容标注
阶段 2：知识内容与考点体系
阶段 3：RAG 检索与跨笔记增强
```

阶段之间存在依赖，禁止三个阶段同时全面铺开。

## 3. 术语与领域对象

以下命名是推荐领域命名。Codex 可以结合当前代码命名习惯调整文件名，但不得合并对象职责。

### 3.1 Note：笔记

原始学习资料的主要载体。

核心要求：

- 拥有稳定 ID。
- 正文更新不改变 Note ID。
- 支持软删除与恢复。
- 能识别当前正文版本或内容哈希。
- 能提供标题层级、正文结构和可定位范围。

### 3.2 ContentAnnotation：重要内容标注

表示用户或 AI 在正文中指出的一段重要内容。

它只回答：

> 哪一段内容值得关注？

建议字段：

```text
id
spaceId
noteId
kind                    // 第一版固定为 important，后续可扩展
sourceMode              // manual / ai
quoteText               // 创建时的原文快照
headingPath             // 所属标题路径
fromPosition            // 编辑器范围起点
toPosition              // 编辑器范围终点
prefixText              // 前文上下文
suffixText              // 后文上下文
anchorFingerprint       // 用于重新定位
noteContentHash         // 创建时对应的笔记内容哈希
status                   // active / stale / archived
createdAt
updatedAt
deletedAt
```

约束：

- 不依赖标题、注释、掌握度等知识内容字段。
- 即使正文后续修改，也必须保留 `quoteText` 快照。
- 定位失败时必须标记为 `stale`，不得静默跳到错误位置。
- 不保存 DOM selector，来源定位必须基于编辑器位置、文本上下文和指纹。

### 3.3 KnowledgeItem：知识内容

表示一个相对稳定、可复用、可关联的知识语义单元。

它回答：

> 这里讲的是什么知识？

建议字段：

```text
id
spaceId
title
canonicalStatement      // 规范化知识陈述
userExplanation         // 用户自己的理解，可为空
knowledgeType           // fact / concept / principle / process / algorithm / formula / comparison / application
importance              // low / medium / high
reviewStatus            // candidate / confirmed / needs_revision / archived
sourceMode              // manual / ai / mixed
createdAt
updatedAt
deletedAt
```

说明：

- 普通学科分类继续使用标签。
- `importance`、`reviewStatus`、`sourceMode` 等稳定业务属性使用结构化字段，不作为普通标签。
- 掌握度不放入该实体，掌握度属于后续训练结果。
- `canonicalStatement` 是 AI 出题和检索时的主要语义内容，不能只保存标题。

### 3.4 KnowledgeEvidence：知识依据关系

用于连接知识内容与标注。

关系必须支持：

```text
一个标注 → 多个知识内容
多个标注 → 一个知识内容
```

建议字段：

```text
id
knowledgeItemId
annotationId
relationType            // supports / example / exception / comparison
createdAt
```

第一版可以只使用 `supports`，但字段需要保留扩展能力。

### 3.5 AssessmentPoint：考点

表示一个可被出题和评测的学习目标。

它回答：

> 学习者需要能够做什么？

建议字段：

```text
id
knowledgeItemId
objective               // 完整的可评测目标
actionVerb              // 说出 / 解释 / 判断 / 比较 / 计算 / 应用 / 分析 / 证明
cognitiveLevel          // remember / understand / apply / analyze
difficultyHint          // basic / medium / advanced
questionTypeSuggestions // 可选题型建议
reviewStatus            // candidate / confirmed / needs_revision / archived
order
createdAt
updatedAt
deletedAt
```

约束：

- `objective` 必须包含可以观察或评测的动作。
- 禁止保存“了解 Prim”“熟悉 TCP”这类不可评测目标。
- 一个知识内容可以拥有多个考点。
- 一个考点后续可以关联多道题目。
- 考点是知识链路与训练链路的连接点。

### 3.6 DocumentChunk：检索切片

用于全文搜索、向量检索和 RAG，不直接等同于知识内容。

建议字段：

```text
id
spaceId
noteId
noteContentHash
headingPath
blockStart
blockEnd
content
contentHash
tokenCount
chunkStrategyVersion
status
createdAt
updatedAt
```

### 3.7 EmbeddingRecord：向量记录

建议字段：

```text
id
resourceType            // document_chunk / knowledge_item / assessment_point
resourceId
contentHash
provider
model
dimensions
embeddingVersion
vector
createdAt
```

不同模型或维度的向量不得混用。

## 4. 模块边界

建议将业务边界设计为模块化单体，不拆微服务。

```text
Knowledge Module
├── Note
├── ContentAnnotation
├── KnowledgeItem
├── KnowledgeEvidence
└── AssessmentPoint

AI Module
├── AIJob
├── PromptTemplate
├── KnowledgeExtraction
└── AssessmentPointGeneration

Retrieval Module
├── DocumentChunk
├── Chunking
├── KeywordSearch
├── Embedding
├── VectorSearch
├── HybridRetrieval
└── Citation

Exam Module（后续）
├── ExamBlueprint
├── Question
├── Exam
└── AnswerRecord
```

依赖方向：

```text
HTTP / Presentation
        ↓
Application Service
        ↓
Domain
        ↓
Repository Interface
        ↓
Infrastructure Adapter
```

AI 与外部模型调用必须经过统一 AI Gateway，不得散落在知识模块的 Service 中。

Retrieval Module 可以读取知识资源的公开查询接口或领域事件，不得直接修改 Knowledge Module 的内部数据。

## 5. 与当前代码的衔接策略

当前项目已经存在知识点实体、来源片段、正文 marker、选区创建、加入已有知识点、右侧知识点面板和相应前后端分层。不得直接推倒重写。

### 5.1 当前能力的推荐映射

```text
现有“知识点来源片段”
→ ContentAnnotation 或 Annotation Source

现有正文 marker 与来源跳转
→ 继续复用，改为标注 marker

现有“知识点”主体
→ 迁移为 KnowledgeItem

现有“加入已有知识点”
→ 改为“将标注关联到已有知识内容”

现有知识点标签
→ 普通标签继续保留
```

### 5.2 旧数据迁移原则

对每个现有知识点执行：

1. 创建一个 `KnowledgeItem`。
2. 将原标题迁移为 `title`。
3. 将原注释或个人理解迁移为 `userExplanation`。
4. 如有可用概括内容，迁移为 `canonicalStatement`；否则使用候选状态并等待补充。
5. 将每个原文来源片段迁移为 `ContentAnnotation`。
6. 创建 `KnowledgeEvidence` 连接知识内容与标注。
7. 暂不自动生成正式考点，可后续批量生成候选考点。
8. 保留旧 ID 映射，保证历史引用和前端状态可迁移。

迁移必须满足：

- 可重复执行且不会产生重复数据。
- 失败可回滚或重新执行。
- 迁移前后数量可核对。
- 不删除旧数据，直到新链路验证通过。

### 5.3 旧标签迁移

- 普通学科标签直接保留。
- 重要程度标签可以转换为 `importance`，验证完成后再移除旧标签关系。
- 旧“未掌握/模糊/已掌握”标签不得直接转换为客观掌握度。
- 如需保留用户主观判断，可迁移为独立的 `selfAssessment` 字段或历史元数据。
- 第一阶段允许只做读取兼容，不急于物理清理旧标签。

### 5.4 兼容期

建议旧知识点 API 保留一个兼容周期：

```text
旧 API
→ Compatibility Adapter
→ 新 Application Service
```

前端迁移完成、测试通过、旧数据验证无误后，再删除旧兼容入口。

## 7. 阶段 1：知识库底座与重要内容标注

阶段 1 只解决：

> 用户可以轻松指出正文中哪些内容重要，并且系统能够稳定保存、展示和重新定位这些内容。

不要求用户立即填写知识标题和精细解释。

### 7.1 阶段 1A：建立 ContentAnnotation 领域模型

后端建议新增：

```text
apps/api/src/modules/knowledge/domain/content-annotation.js
apps/api/src/modules/knowledge/application/annotation-service.js
apps/api/src/modules/knowledge/application/dto/annotation-dto.js
apps/api/src/modules/knowledge/infrastructure/annotation-repository.js
apps/api/src/modules/knowledge/http/annotation-routes.js
```

具体位置应结合现有 repository 组织方式调整，但必须保持现有分层。

主要应用服务：

```text
createAnnotation()
listAnnotationsByNote()
getAnnotation()
updateAnnotation()
deleteAnnotation()
restoreAnnotation()
markAnnotationStale()
reanchorAnnotation()
```

领域约束：

- `quoteText` 不可为空。
- `noteId`、`fromPosition`、`toPosition` 必须合法。
- `fromPosition < toPosition`。
- 手动标注默认 `sourceMode = manual`。
- 删除采用软删除。
- `reanchorAnnotation()` 只能在匹配置信度达到阈值时更新位置，否则保持 stale。

### 7.2 推荐 API

```text
POST   /api/knowledge/annotations
GET    /api/knowledge/annotations?noteId=:noteId
GET    /api/knowledge/annotations/:annotationId
PATCH  /api/knowledge/annotations/:annotationId
DELETE /api/knowledge/annotations/:annotationId
POST   /api/knowledge/annotations/:annotationId/restore
POST   /api/knowledge/annotations/:annotationId/reanchor
```

Controller/HTTP 层只负责：

- 参数绑定与校验。
- 调用 application service。
- 返回统一响应。

禁止在 route 中编写定位、匹配或迁移逻辑。

### 7.3 前端交互

现有“创建知识点”入口改为：

```text
标记为重要内容
```

流程：

```text
用户选中正文
→ 右键或浮动工具条点击“标记为重要内容”
→ 立即创建标注
→ 正文显示轻量 marker
→ 右侧“知识”面板的“标注”视图出现记录
```

第一版创建标注不弹出复杂表单。

右侧面板建议改为：

```text
知识
├── 标注
├── 知识内容
└── 考点
```

阶段 1 只启用“标注”，其他视图可以保留空状态或暂不展示。

标注卡片最小信息：

```text
原文摘要
所属标题
创建方式
状态
定位
删除
后续操作入口：提炼知识
```

### 7.4 前端模块建议

不要继续扩充已经接近行数边界的 `knowledge-api.js`。新增接口前先拆分 service。

推荐：

```text
apps/web/src/services/knowledge/
├── annotation-api.js
├── knowledge-item-api.js        // 阶段 2
├── assessment-point-api.js      // 阶段 2
└── index.js
```

前端建议逐步从旧 `knowledge-point` 命名迁移：

```text
apps/web/src/controllers/knowledge/
├── annotation-controller.js
├── marker-controller.js
└── index.js

apps/web/lib/knowledge/
├── annotation-state.js
├── annotation-selection.js
├── annotation-renderers.js
└── annotation-filtering.js
```

迁移期间可以保留旧聚合入口做适配，不得一次性改动所有 import 后再统一排错。

### 7.5 正文定位策略

每条标注同时保存：

1. 编辑器位置范围。
2. 原文快照。
3. 标题路径。
4. 前后文。
5. 内容哈希或指纹。

定位顺序建议：

```text
当前内容哈希一致
→ 直接使用位置范围

内容已变化
→ 在原标题范围内匹配 quoteText

精确匹配失败
→ 使用 prefix + quote + suffix 进行上下文匹配

仍失败
→ 标记 stale，要求用户手动重新选择
```

禁止在低置信度情况下自动跳转到错误文本。

### 7.6 阶段 1 测试

后端：

- annotation domain 正常与异常输入。
- repository 创建、更新、软删除、恢复。
- 同一笔记标注列表。
- 重新定位成功与失败。
- API 参数校验与错误码。
- 旧来源片段迁移的幂等性。

前端：

- 从有效选区构造 annotation input。
- 空选区不得创建。
- marker 与 annotation id 正确关联。
- 点击正文 marker 聚焦右侧卡片。
- 点击卡片定位正文。
- stale 状态正确展示。
- 删除后 marker 消失。
- 页面刷新后标注仍存在。

### 7.7 阶段 1 验收标准

- 用户只需选择正文并点击一次，即可完成标注。
- 标注刷新后不会丢失。
- 正文与右侧卡片支持双向定位。
- 一个笔记可以保存多个重叠或相邻标注。
- 正文轻微修改后能尝试重新定位。
- 定位失败会明确显示失效状态。
- 旧知识点来源数据迁移后无丢失。
- 未引入 AI、向量数据库和试卷功能。

## 8. 阶段 2：知识内容与考点体系

阶段 2 解决：

> 将宽泛标注提炼为规范知识内容，并进一步拆解成可评测考点。

### 8.1 阶段 2A：KnowledgeItem

后端建议新增或从旧知识点实体迁移：

```text
domain/knowledge-item.js
application/knowledge-item-service.js
application/dto/knowledge-item-dto.js
infrastructure/knowledge-item-repository.js
http/knowledge-item-routes.js
```

主要应用服务：

```text
createKnowledgeItem()
updateKnowledgeItem()
listKnowledgeItems()
getKnowledgeItem()
archiveKnowledgeItem()
confirmKnowledgeItem()
linkAnnotation()
unlinkAnnotation()
mergeKnowledgeItems()
```

领域约束：

- `title` 和 `canonicalStatement` 在正式确认时必须存在。
- 候选态可以允许字段不完整。
- 正式知识内容至少拥有一个证据来源；纯手工创建时可先为草稿，但确认前必须补来源或明确标记为用户自建。
- 合并知识内容时必须迁移证据、考点和标签关系。
- 删除优先归档或软删除。

### 8.2 阶段 2B：KnowledgeEvidence

实现多对多来源关系：

```text
Annotation A ─┬→ Knowledge Item 1
              └→ Knowledge Item 2

Annotation B ───→ Knowledge Item 1
```

前端支持：

- 从标注创建新的知识内容。
- 将标注关联到已有知识内容。
- 从知识内容移除某个来源，但不删除原标注。
- 查看一个知识内容的所有来源。
- 点击来源回到原文。

“删除知识内容”与“删除标注”必须严格区分。

### 8.3 阶段 2C：AssessmentPoint

后端建议新增：

```text
domain/assessment-point.js
application/assessment-point-service.js
application/dto/assessment-point-dto.js
infrastructure/assessment-point-repository.js
http/assessment-point-routes.js
```

主要应用服务：

```text
createAssessmentPoint()
updateAssessmentPoint()
listAssessmentPointsByKnowledgeItem()
```

## 9. AI 提炼流程

### 9.1 所有 AI 调用统一走 AI Module

建议结构：

```text
Knowledge Application Service
        ↓
AI Job Service
        ↓
Knowledge Extraction Use Case
        ↓
AI Gateway
        ↓
Provider Adapter
```

知识模块不得直接依赖 OpenAI、Claude、Qwen 等具体供应商 SDK。

### 9.2 AIJob

建议字段：

```text
id
jobType
status                  // pending / running / succeeded / failed / cancelled
inputReferences
promptVersion
provider
model
resultReference
errorCode
errorMessage
createdAt
startedAt
finishedAt
```

规则：

- 外部模型调用不得放在数据库事务中。
- 批量分析必须使用 Job 模式。
- 初期可以使用本地执行器，但 API 契约仍按 Job 设计，以便后续接入队列。
- 任务必须支持失败状态和重试，不得只返回 null。
- 同一请求应有幂等键，避免用户重复点击产生重复候选。

### 9.3 单个标注提炼

流程：

```text
标注
→ 获取原文、标题路径和相邻上下文
→ AI 生成 1～N 个知识内容候选
→ JSON Schema 校验
→ 保存 candidate
→ 用户编辑、确认、合并或拒绝
```

建议 AI 输出：

```json
{
  "knowledgeCandidates": [
    {
      "title": "Prim 算法的选边规则",
      "canonicalStatement": "……",
      "knowledgeType": "algorithm",
      "importance": "high",
      "sourceAnnotationIds": ["annotation-id"],
      "confidence": 0.91
    }
  ]
}
```

`confidence` 仅用于排序和提示，不作为自动确认依据。

### 9.4 考点生成

流程：

```text
已确认或候选知识内容
→ AI 生成考点候选
→ 结构校验
→ 用户确认
```

建议输出：

```json
{
  "assessmentPointCandidates": [
    {
      "objective": "能够解释 Prim 每轮选择跨割最小边的原因",
      "actionVerb": "解释",
      "cognitiveLevel": "understand",
      "difficultyHint": "medium",
      "questionTypeSuggestions": ["short_answer", "judgment"]
    }
  ]
}
```

### 9.5 整篇笔记自动分析

流程：

```text
用户点击“分析本篇笔记”
→ 按标题结构划分分析单元
→ 创建 AIJob
→ 分批提取重要区域和知识内容候选
→ 跨章节去重
→ 生成考点候选
→ 进入批量审核界面
```

第一版不依赖向量数据库。

分析上下文优先使用：

- 当前章节内容。
- 标题路径。
- 相邻章节摘要。
- 笔记元数据。

禁止为了整篇分析，直接将任意长度的完整笔记一次性塞给模型。

### 9.6 AI 输出校验

每个 AI 用例必须包含：

- JSON Schema 或等价结构校验。
- 枚举校验。
- 长度限制。
- 空字段检查。
- 来源 ID 合法性检查。
- 重复候选检查。
- 错误转换与重试策略。
- Prompt 版本记录。

结构校验失败时，不得写入正式数据。

## 13. 阶段 3：RAG 检索与跨笔记增强

阶段 3 解决：

> 当笔记数量增多后，AI 能够从正确范围内找到相关原文、知识内容和考点，并提供可追溯引用。

### 13.1 阶段 3A：结构化切片

切片优先遵循 Markdown 结构：

```text
标题层级
段落
列表
引用
代码块
表格
公式
```

策略：

1. 先按标题章节分组。
2. 章节过长时再按块边界切分。
3. 不在代码块、表格和公式中间随意切断。
4. 每个切片保存完整标题路径。
5. 必要时保留少量重叠上下文。
6. 使用 `chunkStrategyVersion` 标记策略版本。
7. 使用 `contentHash` 支持增量更新。

禁止只按照固定字符数粗暴切分全部内容。

### 13.2 阶段 3B：关键词检索

先在现有存储能力上完成：

- 笔记标题搜索。
- 正文关键词搜索。
- 标题路径搜索。
- KnowledgeItem 搜索。
- AssessmentPoint 搜索。
- 标签与空间过滤。
- 资源类型过滤。

关键词检索是混合检索的基础，不能因为引入向量检索而废弃。

### 13.3 阶段 3C：向量检索

生产环境推荐：

```text
PostgreSQL + pgvector
```

但当前项目仍可能使用本地 JSON 存储，因此必须通过 repository interface 隔离实现。

禁止：

- 直接把大量向量数组写入现有 `knowledge-base.json`。
- 在业务 service 中写死 pgvector SQL。
- 为了 RAG 一次性重写全部知识库持久化。

推荐接口：

```text
EmbeddingProvider
EmbeddingRepository
VectorSearchRepository
```

允许实现：

```text
Keyword-only adapter        // 阶段 3A/3B
PgVector adapter            // 正式向量实现
Test in-memory adapter      // 测试
```

### 13.4 索引对象

第一批只索引：

```text
DocumentChunk
Confirmed KnowledgeItem
Confirmed AssessmentPoint
```

默认不索引：

- candidate。
- archived。
- deleted。
- stale 且无法确认来源的内容。
- 纯 UI 状态。

### 13.5 增量索引

笔记保存后：

```text
计算 noteContentHash
→ 重新生成该笔记切片
→ 比较 chunk contentHash
→ 保留未变化切片
→ 新增或更新变化切片
→ 删除失效切片
→ 只为变化资源重新生成 embedding
```

知识内容或考点更新后同理。

软删除资源必须立即从检索结果中过滤，物理删除时再清理向量记录。

### 13.6 混合检索

推荐流程：

```text
用户查询
→ 查询标准化
→ 结构化过滤
→ 关键词召回
→ 向量召回
→ 合并去重
→ 可选重排序
→ 返回引用
```

过滤条件可以包括：

```text
spaceId
noteId
folderId
tagIds
resourceType
reviewStatus
knowledgeType
timeRange
```

### 13.7 引用模型

每个检索结果至少返回：

```text
resourceType
resourceId
noteId
headingPath
quoteText
score
matchedBy           // keyword / vector / hybrid
location
```

AI 问答必须返回可点击引用。

引用点击后：

- 原文切片跳转到对应笔记位置。
- KnowledgeItem 跳转到知识内容卡片。
- AssessmentPoint 跳转到考点卡片。

### 13.8 RAG 问答

流程：

```text
用户问题
→ 混合检索
→ 选择上下文
→ AI 基于上下文回答
→ 区分“来自笔记”与“AI 补充”
→ 返回引用
```

回答规则：

- 无足够依据时明确说明资料不足。
- 不得伪造笔记引用。
- 引用必须来自实际检索结果。
- 用户可选择“仅根据我的笔记回答”或“允许 AI 补充”。

### 13.9 RAG 对知识链路的增强

完成基础问答后，再按顺序加入：

1. 新知识内容创建前的相似项查重。
2. 为知识内容补充其他笔记来源。
3. 查找前置知识、相关知识和易混知识。
4. 跨笔记生成知识内容候选。
5. 跨笔记生成考点。
6. 按主题或标签生成出题范围。
7. 为题目答案提供来源校验。
8. 错题后推荐复习原文。

## 16. 数据一致性与生命周期

### 16.1 删除关系

必须明确区分：

```text
删除标注
删除知识内容
删除考点
删除笔记
```

规则建议：

- 删除标注：移除正文 marker，同时删除或失效其 KnowledgeEvidence；不自动删除知识内容。
- 删除知识内容：归档其考点和证据关系；不删除原文标注。
- 删除考点：不影响知识内容。
- 删除笔记：标注和原文切片软删除或失效；由这些标注支持的知识内容进入“来源不足”状态，不应静默删除。
- 恢复笔记：重新检查标注定位并恢复索引。

### 16.2 来源不足

知识内容的全部来源失效后：

```text
sourceStatus = missing
```

前端提示用户：

- 补充来源。
- 重新绑定原文。
- 保留为用户自建知识。
- 归档。

### 16.3 幂等性

以下操作必须幂等：

- 旧知识点数据迁移。
- 笔记切片重建。
- embedding 生成。
- AI 批量提炼请求。
- 用户重复点击确认。
- Job 重试。

推荐使用：

```text
业务唯一键
contentHash
idempotencyKey
版本字段
```

## 20. 代码与文件拆分要求

### 20.1 后端

- route/controller 不写业务判断。
- application service 不依赖 HTTP 对象。
- domain 不依赖存储、AI SDK 或环境变量。
- repository 负责持久化。
- AI provider 与 embedding provider 使用 adapter。
- 复杂流程拆为步骤函数。
- 禁止循环查库或循环调用外部 API。
- 新实体按 domain/application/infrastructure/http/test 补齐。

### 20.2 前端

- API 请求进入 `src/services`。
- DOM 与业务流程编排进入 `src/controllers`。
- 纯状态、纯规则和 HTML renderer 进入 `lib`。
- 事件 binder 只转发事件，不写 API 或复杂业务。
- 不把新逻辑回填到 `client.js`。
- 新增固定视觉规则进入 styles 与 tokens。
- `knowledge-api.js` 已接近边界，新增功能前优先拆分。

### 20.3 行数

- 普通业务源文件超过 250 行必须自查职责。
- 超过 400～500 行且职责混杂时优先拆分。
- 不为了压行数机械拆文件，拆分依据必须是职责边界。
- 聚合入口可以略长，但不得混入业务逻辑。

## 24. 当前阶段明确不做

- 不先做多 Agent 自动学习系统。

- 不先做跨全库自动生成大型试卷。
- 不先做独立向量数据库集群。
- 不先做微服务拆分。
- 不让 AI 自动修改正式知识。
- 不为了 RAG 重写整个现有项目。
- 不把知识内容、考点、掌握度重新塞回标签体系。
- 不把 DocumentChunk 当成知识内容。
- 不把正文标注重新叫作考点或知识点。

## 25. 最终产品流程

### 手动模式

```text
用户阅读笔记
→ 标注重要内容
→ AI 提炼知识内容候选
→ 用户确认
→ AI 生成考点候选
→ 用户确认
→ 选择考点生成试卷
```

### 自动模式

```text
用户选择一篇笔记或一个范围
→ AI 分析原文
→ 生成标注、知识内容和考点候选
→ 用户批量审核
→ 形成可出题范围
```

### RAG 增强模式

```text
用户提问或选择主题
→ 混合检索相关原文、知识内容和考点
→ AI 回答或生成候选
→ 展示真实引用
→ 用户确认后进入正式知识链路
```

---

## 26. 最终架构原则

整条知识链路必须始终保持以下职责：

```text
笔记负责承载原始资料
标注负责指出重要位置
知识内容负责表达知识本身
考点负责定义学习目标
RAG 负责找到相关材料
试卷负责组织评测
答题与批改负责产生学习证据
掌握度负责反映学习结果
```

任何后续功能设计都不得重新混淆这些对象。

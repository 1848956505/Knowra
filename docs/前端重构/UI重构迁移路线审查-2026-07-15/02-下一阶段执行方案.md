# Knowra UI 重构下一阶段执行方案

## 1. 下一步应该做什么

下一步不是修改正式项目，而是先完成一次 **demo 设计基线收敛**。

这次收敛的目标不是让 demo 看起来“像最终产品”，而是消除会造成正式迁移返工的结构不确定性。完成后，立即进入正式项目的第一条纵向切片，不继续在 demo 中无限打磨。

## 2. 阶段 A：建立设计契约

### A1. 固定设计 token

在 demo 中统一并集中管理：

- 颜色：纸张背景、正文、弱化文字、钴蓝、分隔线、状态色；
- 字体：中文正文字体、标题字体、编号窄体；
- 字号与行高：页面标题、模块编号、正文、说明、计数；
- 间距：4px 或 8px 基础步长及常用组合；
- 线条：普通分隔线、强调线、选中线；
- 布局：左栏、主区、右栏宽度及最小宽度；
- 断点：优先确定 `1440px` 和 `1280px`。

产物：demo 中只有一套 token 来源，不再在各组件内散落重复数值。

### A2. 固定四种壳层状态

需要完成并截图确认：

1. 未选择模块的全局索引；
2. 选择 01 资料库后的目录态；
3. 资料索引与详情栏；
4. 资料编辑器与知识边注。

四种状态必须共享同一套左侧栏和栅格规则，不能各自调整出互不兼容的尺寸。

### A3. 加入极端数据

demo 至少补充以下测试数据：

- 24～32 个汉字的长文件夹名；
- 3～4 层目录；
- 空目录；
- 20 个以上同级文件；
- 8 个打开标签页；
- 超长资料标题；
- 右侧面板内容超出一屏；
- 无标签、无关联、无附件等空状态。

### A4. 补齐基础交互状态

- hover、active、selected、disabled；
- 明显且统一的 `focus-visible`；
- 目录展开和收起；
- 右侧面板展开和收起；
- 长文本省略及完整名称查看方式；
- 键盘 Tab 顺序的基本验证。

## 3. 阶段 A 的验收门槛

同时满足以下条件后，停止打磨 demo：

- `1440 × 1024` 与 `1280 × 800` 均无非预期横向溢出；
- 左侧栏固定，目录独立滚动，底部模块切换始终可见；
- 超长标题、多标签和右栏同时出现时，编辑区仍可用；
- 四种壳层状态共享同一套 token 与布局规则；
- 长名称、深层目录、空状态和大量文件都有明确处理；
- 键盘焦点可见，图标按钮有文本标签；
- 《Knowra 左侧导航栏设计说明》与 demo 行为一致；
- 不再存在会影响 DOM 主结构的未决方案。

## 4. 阶段 B：正式项目第一条纵向切片

验收阶段 A 后，在正式项目中只迁移以下路径：

> 全局壳层 → 01 资料库 → 文件夹目录 → 打开资料 → Milkdown 编辑 → 保存 → 返回目录

实施原则：

- 保留现有 API、state、controller、renderer 和 Milkdown；
- 只替换壳层 DOM、渲染结构和 CSS，不迁移到 React；
- 新视觉 token 先映射到正式项目的 `tokens.css`；
- 新左侧栏落在现有 shell/navigation renderer 中；
- 每完成一个状态就恢复现有功能并跑对应测试；
- 旧 UI 在切片完成前保持可回退，不进行一次性大删除。

功能对照至少覆盖：

- 文件夹展开、选择、新建、重命名、删除和拖拽；
- 文件打开、多标签和关闭；
- 编辑、自动保存、预览和状态栏；
- 标签、附件、边注和大纲；
- 导入导出、查找替换、图片和表格；
- 右键菜单和快捷键。

## 5. 阶段 C：扩展到其他状态

第一条纵向切片通过功能回归与视觉 QA 后，再依次处理：

1. 搜索与搜索结果；
2. 收藏、最近和回收站；
3. 空状态、错误状态和加载状态；
4. 02～05 模块的正式二级目录；
5. 小屏幕和移动端策略。

## 6. 当前暂不处理

- 不全面重写正式项目；
- 不把正式项目迁移到 React；
- 不提前细化 02～05 模块的所有页面；
- 不在 demo 中复刻完整 Milkdown 功能；
- 不围绕单个页面做脱离整体栅格的像素级微调；
- 不删除现有功能来迁就新视觉布局。

## 7. 推荐执行顺序

1. 确认《03-颗粒度对齐清单》的三个默认决策；
2. 在测试 demo 中完成阶段 A；
3. 输出两种桌面宽度的四状态截图矩阵；
4. 依据验收门槛完成一次设计审查；
5. 进入正式项目的第一条纵向切片；
6. 每完成一个切片，执行功能回归与视觉对比。

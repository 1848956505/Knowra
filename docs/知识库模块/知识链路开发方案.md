# 知识链路开发方案（Codex 执行版）

> 文档定位：本文件用于指导 Codex 在现有项目中逐阶段开发“知识库底座 → 知识提炼与考点体系 → RAG 知识库”整条知识链路。
>
> 本文件是实施方案，不是一次性代码生成清单。Codex 必须按阶段执行、逐步验证，不得将全部功能塞入一次改动。
>
> 建议存放路径：`docs/知识链路开发方案.md`

---

## 1. 项目目标

本项目的“知识”能力不只是保存笔记，也不只是提供 AI 问答，而是要将原始学习资料逐步转化为可训练、可评测、可追踪的学习结构。

完整链路如下：

```text
笔记原文
→ 重要内容标注
→ 知识内容
→ 考点
→ 试卷蓝图
→ 题目与试卷
→ 作答与批改
→ 掌握度
```

RAG 作为检索增强层，贯穿原文、知识内容和考点：

```text
笔记原文 / 知识内容 / 考点
→ 结构化切片与索引
→ 关键词检索 + 向量检索 + 结构化过滤
→ AI 问答、查重、关联、跨笔记出题与复习推荐
```

本方案聚焦“知识”链路本身，试卷、答题、批改和掌握度只定义接口边界，不在本阶段一次性全部实现。

## 2. 核心设计结论

### 2.1 不再把正文选区直接等同于知识点

正文中被选中的内容通常宽泛、上下文依赖较强，适合表达“这里重要”，但不一定是一个边界清晰、可直接出题的知识单元。

因此必须区分：

```text
标注：哪里重要
知识内容：这里讲了什么
考点：学习者需要会什么
```

### 2.2 知识内容与考点必须分离

- **知识内容（Knowledge Item）**：对知识本身的规范化表达。
- **考点（Assessment Point）**：围绕知识内容定义的可评测学习目标。

示例：

```text
知识内容：
Prim 算法每次选择一条连接已访问集合和未访问集合的最小权边，
逐步构造最小生成树。

考点：
1. 能够说明 Prim 算法解决的问题。
2. 能够解释 Prim 每轮的选边规则。
3. 能够手算 Prim 的执行过程。
4. 能够比较 Prim 与 Dijkstra 的贪心目标。
```

### 2.3 RAG 不替代结构化知识模型

RAG 负责“找到相关材料”，知识内容与考点负责“组织知识并定义学习目标”。

```text
RAG = 找材料
知识内容 = 组织材料
考点 = 定义评测目标
试卷蓝图 = 规划考试
题目 = 执行评测
```

### 2.4 AI 结果必须先进入候选态

AI 自动生成的知识内容、考点、关联关系和题目不得直接成为正式数据。

统一流程：

```text
AI 生成
→ 结构校验
→ 候选数据
→ 用户确认或修改
→ 正式数据
```

### 2.5 同时设计，分阶段开发

开发顺序调整为：

```text
阶段 0：旧知识点实现退役审计
阶段 1：清除旧知识点业务链路并建立 Annotation 最小底座
阶段 2：KnowledgeItem 与 AssessmentPoint
阶段 3：RAG 检索与跨笔记增强
```

阶段之间存在严格依赖，禁止同时全面铺开。

现有知识点代码不再被视为可直接演化的正式底座。它只能作为旧原型和编辑器接入参考。新实现不得继续以旧 `KnowledgePoint` 聚合为中心。

## 3. 术语与领域对象

以下命名是推荐领域命名。Codex 可以结合当前代码命名习惯调整文件名，但不得合并对象职责。

### 3.1 Note：笔记

原始学习资料的主要载体。

核心要求：

- 拥有稳定 ID。
- 正文更新不改变 Note ID。
- 支持软删除与恢复。
- 能识别当前正文版本或内容哈希。
- 能提供标题层级、正文结构和可定位范围。

### 3.2 ContentAnnotation：重要内容标注

表示用户或 AI 在正文中指出的一段重要内容。

它只回答：

> 哪一段内容值得关注？

建议字段：

```text
id
spaceId
noteId
kind                    // 第一版固定为 important，后续可扩展
sourceMode              // manual / ai
quoteText               // 创建时的原文快照
headingPath             // 所属标题路径
fromPosition            // 编辑器范围起点
toPosition              // 编辑器范围终点
prefixText              // 前文上下文
suffixText              // 后文上下文
anchorFingerprint       // 用于重新定位
noteContentHash         // 创建时对应的笔记内容哈希
status                   // active / stale / archived
createdAt
updatedAt
deletedAt
```

约束：

- 不依赖标题、注释、掌握度等知识内容字段。
- 即使正文后续修改，也必须保留 `quoteText` 快照。
- 定位失败时必须标记为 `stale`，不得静默跳到错误位置。
- 不保存 DOM selector，来源定位必须基于编辑器位置、文本上下文和指纹。

### 3.3 KnowledgeItem：知识内容

表示一个相对稳定、可复用、可关联的知识语义单元。

它回答：

> 这里讲的是什么知识？

建议字段：

```text
id
spaceId
title
canonicalStatement      // 规范化知识陈述
userExplanation         // 用户自己的理解，可为空
knowledgeType           // fact / concept / principle / process / algorithm / formula / comparison / application
importance              // low / medium / high
reviewStatus            // candidate / confirmed / needs_revision / archived
sourceMode              // manual / ai / mixed
createdAt
updatedAt
deletedAt
```

说明：

- 普通学科分类继续使用标签。
- `importance`、`reviewStatus`、`sourceMode` 等稳定业务属性使用结构化字段，不作为普通标签。
- 掌握度不放入该实体，掌握度属于后续训练结果。
- `canonicalStatement` 是 AI 出题和检索时的主要语义内容，不能只保存标题。

### 3.4 KnowledgeEvidence：知识依据关系

用于连接知识内容与标注。

关系必须支持：

```text
一个标注 → 多个知识内容
多个标注 → 一个知识内容
```

建议字段：

```text
id
knowledgeItemId
annotationId
relationType            // supports / example / exception / comparison
createdAt
```

第一版可以只使用 `supports`，但字段需要保留扩展能力。

### 3.5 AssessmentPoint：考点

表示一个可被出题和评测的学习目标。

它回答：

> 学习者需要能够做什么？

建议字段：

```text
id
knowledgeItemId
objective               // 完整的可评测目标
actionVerb              // 说出 / 解释 / 判断 / 比较 / 计算 / 应用 / 分析 / 证明
cognitiveLevel          // remember / understand / apply / analyze
difficultyHint          // basic / medium / advanced
questionTypeSuggestions // 可选题型建议
reviewStatus            // candidate / confirmed / needs_revision / archived
order
createdAt
updatedAt
deletedAt
```

约束：

- `objective` 必须包含可以观察或评测的动作。
- 禁止保存“了解 Prim”“熟悉 TCP”这类不可评测目标。
- 一个知识内容可以拥有多个考点。
- 一个考点后续可以关联多道题目。
- 考点是知识链路与训练链路的连接点。

### 3.6 DocumentChunk：检索切片

用于全文搜索、向量检索和 RAG，不直接等同于知识内容。

建议字段：

```text
id
spaceId
noteId
noteContentHash
headingPath
blockStart
blockEnd
content
contentHash
tokenCount
chunkStrategyVersion
status
createdAt
updatedAt
```

### 3.7 EmbeddingRecord：向量记录

建议字段：

```text
id
resourceType            // document_chunk / knowledge_item / assessment_point
resourceId
contentHash
provider
model
dimensions
embeddingVersion
vector
createdAt
```

不同模型或维度的向量不得混用。

## 4. 模块边界

建议将业务边界设计为模块化单体，不拆微服务。

```text
Knowledge Module
├── Note
├── ContentAnnotation
├── KnowledgeItem
├── KnowledgeEvidence
└── AssessmentPoint

AI Module
├── AIJob
├── PromptTemplate
├── KnowledgeExtraction
└── AssessmentPointGeneration

Retrieval Module
├── DocumentChunk
├── Chunking
├── KeywordSearch
├── Embedding
├── VectorSearch
├── HybridRetrieval
└── Citation

Exam Module（后续）
├── ExamBlueprint
├── Question
├── Exam
└── AnswerRecord
```

依赖方向：

```text
HTTP / Presentation
        ↓
Application Service
        ↓
Domain
        ↓
Repository Interface
        ↓
Infrastructure Adapter
```

AI 与外部模型调用必须经过统一 AI Gateway，不得散落在知识模块的 Service 中。

Retrieval Module 可以读取知识资源的公开查询接口或领域事件，不得直接修改 Knowledge Module 的内部数据。

## 5. 现有代码审计结论与退役决策

本节结论基于当前提供的后端、前端核心、Milkdown 集成和顶层装配代码，而不是仅根据项目结构文档推测。

### 5.1 总体结论

当前知识点实现是一套未完成、未投入真实使用的原型链路，不适合继续承担新知识体系的底座。

处理原则改为：

```text
不迁移旧业务模型
不保留旧 API 兼容期
不把旧 KnowledgePoint 改名后继续使用

先提取可复用的通用编辑器能力
再删除旧知识点业务链路
最后从 ContentAnnotation 重新建立底座
```

仓库当前 `knowledgePoints` 没有真实用户数据，因此默认不制作正式业务数据迁移脚本。Codex 仍需检查根级运行时数据、缓存和 mock，确认只有空数组或测试数据后方可清理。

### 5.2 后端旧实现判断

当前后端对象为：

```text
KnowledgePoint
├── title
├── comment
├── tags
├── note relations
└── one or more KnowledgePointSource
```

该设计与新领域模型存在根本差异：

1. `normalizeSources()` 强制知识点创建时至少包含一个来源。
2. 选区来源和知识主体在同一次创建中产生。
3. `title + comment + tags + sources` 被组合成一个聚合。
4. 掌握程度、重要程度和用途被固化为系统标签组。
5. 没有 `ContentAnnotation / KnowledgeItem / KnowledgeEvidence / AssessmentPoint` 的职责分离。
6. repository、service、route 和测试均围绕旧聚合设计。

因此以下后端文件列入退役范围：

```text
apps/api/src/modules/knowledge/domain/knowledge-point.js
apps/api/src/modules/knowledge/application/knowledge-point-service.js
apps/api/src/modules/knowledge/application/knowledge-point-tag-groups.js
apps/api/src/modules/knowledge/infrastructure/knowledge-point-repository.js
apps/api/src/modules/knowledge/http/knowledge-point-routes.js
apps/api/test/knowledge-point-service.test.js
apps/api/test/knowledge-point-http.test.js
```

同时从以下装配位置移除旧依赖：

```text
apps/api/src/modules/knowledge/index.js
知识库 route 聚合入口
文件数据存储的旧 knowledgePoints / sources / relations / tagGroups 集合
导入导出 schema 中仅服务旧知识点的字段
```

旧的通用 `createCollectionRepository()` 思路可以参考，但不要为了复用而保留整个 `knowledge-point-repository.js`。新 Annotation repository 应具有独立命名、独立集合和独立查询契约。

### 5.3 前端旧业务判断

当前前端流程是：

```text
getSelectionSnapshot()
→ buildKnowledgePointInputFromSelection()
→ 自动从原文截取标题
→ 直接创建 KnowledgePoint
→ 更新 state.knowledgePoints
→ 同步 marker
→ 展开右侧知识点卡片
```

该流程与新产品定义冲突，因为用户选中的宽泛内容应先成为标注，而不是立即成为知识内容。

以下前端业务文件列入退役范围：

```text
apps/web/src/controllers/knowledge-point-controller.js
apps/web/src/controllers/knowledge-point/state-controller.js
apps/web/src/controllers/knowledge-point/selection-controller.js
apps/web/src/controllers/knowledge-point/mutation-controller.js
apps/web/src/controllers/knowledge-point/marker-controller.js

apps/web/lib/knowledge-points/state.js
apps/web/lib/knowledge-points/selection.js
apps/web/lib/knowledge-points/form.js
apps/web/lib/knowledge-points/filtering.js
apps/web/lib/knowledge-points/panel-utils.js
apps/web/lib/knowledge-points/panel.js
apps/web/lib/knowledge-points/panel-renderers.js

apps/web/src/services/knowledge-api/knowledge-point-service.js
```

旧 `local/cache` 分支只通过 `createLocalKnowledgePointAggregate()` 等函数更新浏览器内存集合，没有形成清晰可靠的持久化链路。新 Annotation 功能不得延续这套“API 模式一套、本地模式另一套临时业务实现”的分叉。当前阶段应以真实后端持久化为唯一验收链路；缓存只用于读取降级，不作为正式写入源。

### 5.4 顶层状态和事件清理范围

从 `app-state.js` 删除旧知识点业务状态：

```text
knowledgePoints
allKnowledgePoints
knowledgePointTagGroups
knowledgePointFilters
knowledgePointAttachComposer
expandedKnowledgePointIds
knowledgePointEditing
```

阶段 1 重新建立最小 Annotation 状态：

```text

annotationLoadState
focusedAnnotationId
expandedAnnotationIds
annotationFilter
```

其中：

- `annotations` 是业务数据快照。
- `focusedAnnotationId`、`expandedAnnotationIds`、`annotationFilter` 是纯视图状态。
- 不再维护 `allKnowledgePoints + knowledgePoints` 两套容易失同步的集合。

从侧栏和事件层移除所有 `data-knowledge-point-*` 分支，包括过滤、编辑、挂载、来源移除、删除和来源跳转。

需要修改：

```text
apps/web/lib/events/aside-events/click.js
apps/web/lib/events/aside-events/input.js
apps/web/lib/events/aside-events/forms.js
apps/web/lib/events/editor-content-events.js
apps/web/src/controllers/sidebar-controller.js
apps/web/src/controllers/app-controller-registry.js
apps/web/src/controllers/event-bindings-controller.js
```

侧栏通用 tab、附件、标签和大纲逻辑必须保留，不得因删除知识点分支而改坏。

### 5.5 可以保留并抽象的通用能力

以下能力与旧知识点业务相对解耦，应优先保留：

#### 必须保留

```text
Milkdown Host 的 getSelectionSnapshot()
通用编辑器 selection controller
编辑器 Host 生命周期
侧栏通用 tab 与独立滚动结构
统一 API client
controller 注册与事件 binder 架构
```

#### 允许重构后保留

当前：

```text
apps/web/lib/editor/milkdown/host/knowledge-point-source-controller.js
apps/web/lib/editor/milkdown/plugins/knowledge-point-highlight-plugin.js
```

建议重构为：

```text
apps/web/lib/editor/milkdown/host/annotation-source-controller.js
apps/web/lib/editor/milkdown/plugins/annotation-highlight-plugin.js
```

保留的通用能力仅限：

```text
向编辑器注入一组来源范围
创建 Decoration
点击 marker 派发 annotation id
根据 annotation id 选择正文范围
```

必须删除或修改的旧语义：

```text
knowledgePointId
knowledge-point-marker-click
data-knowledge-point-source-id
data-knowledge-point-id
setKnowledgePointSources()
selectKnowledgePointSource()
```

替换为：

```text
annotationId
annotation-marker-click
data-annotation-id
setAnnotations()
selectAnnotation()
```

### 5.6 旧 marker 插件不能原样复用

当前范围解析顺序为：

```text
anchor 范围
→ offset 范围
→ collectDocumentTextMatches(...)[0]
```

最后一步直接选择第一个文本匹配。当相同文本在正文中重复出现时，可能定位到错误位置；同时无法向业务层报告 `stale`。

新实现必须：

1. 优先验证保存的 ProseMirror 范围。
2. 使用 `quoteText + prefixText + suffixText + headingPath` 消歧。
3. 计算匹配置信度。
4. 唯一高置信度匹配才自动重定位。
5. 多个同分匹配或低置信度匹配返回 unresolved。
6. 由 application/controller 将标注标记为 `stale`。
7. 禁止静默使用第一个匹配。
8. marker 插件只负责渲染已解析范围，不承担持久化或业务状态变更。

### 5.7 不需要旧数据兼容期

由于没有真实知识点数据，本次改造默认采用删除式替换：

```text
旧 API 下线
旧 state 删除
旧 UI 删除
旧测试删除或改写
新 Annotation API 上线
新 Annotation UI 接入
```

但删除前必须完成一次全局引用扫描，检查：

```text
knowledge-point
knowledgePoint
knowledgePoints
knowledge-point-marker-click
data-knowledge-point
setKnowledgePointSources
selectKnowledgePointSource
```

完成新链路并通过回归测试后，上述旧业务标识在人工源码中应为零；历史文档和工程变更日志中的文字记录除外。

## 7. 阶段 1：删除旧业务并建立 ContentAnnotation 底座

### 7.11 草稿一致性

当前编辑器存在自动保存和 `draftMarkdown`。创建标注前必须处理版本一致性：

2. `noteContentHash` 必须基于当前编辑器内容，而不是旧后端正文。
3. 推荐先执行一次当前草稿持久化，再创建 Annotation。
4. 若保存失败，不创建标注。
5. 后端应验证请求中的 noteContentHash 与当前笔记版本；冲突时返回明确错误并要求重新获取选区。

## 9. AI 提炼流程

### 9.1 所有 AI 调用统一走 AI Module

建议结构：

```text
Knowledge Application Service
        ↓
AI Job Service
        ↓
Knowledge Extraction Use Case
        ↓
AI Gateway
        ↓
Provider Adapter
```

知识模块不得直接依赖 OpenAI、Claude、Qwen 等具体供应商 SDK。

### 9.2 AIJob

建议字段：

```text
id
jobType
status                  // pending / running / succeeded / failed / cancelled
inputReferences
promptVersion
provider
model
resultReference
errorCode
errorMessage
createdAt
startedAt
finishedAt
```

规则：

- 外部模型调用不得放在数据库事务中。
- 批量分析必须使用 Job 模式。
- 初期可以使用本地执行器，但 API 契约仍按 Job 设计，以便后续接入队列。
- 任务必须支持失败状态和重试，不得只返回 null。
- 同一请求应有幂等键，避免用户重复点击产生重复候选。

### 9.3 单个标注提炼

流程：

```text
标注
→ 获取原文、标题路径和相邻上下文
→ AI 生成 1～N 个知识内容候选
→ JSON Schema 校验
→ 保存 candidate
→ 用户编辑、确认、合并或拒绝
```

建议 AI 输出：

```json
{
  "knowledgeCandidates": [
    {
      "title": "Prim 算法的选边规则",
      "canonicalStatement": "……",
      "knowledgeType": "algorithm",
      "importance": "high",
      "sourceAnnotationIds": ["annotation-id"],
      "confidence": 0.91
    }
  ]
}
```

`confidence` 仅用于排序和提示，不作为自动确认依据。

### 9.4 考点生成

流程：

```text
已确认或候选知识内容
→ AI 生成考点候选
→ 结构校验
→ 用户确认
```

建议输出：

```json
{
  "assessmentPointCandidates": [
    {
      "objective": "能够解释 Prim 每轮选择跨割最小边的原因",
      "actionVerb": "解释",
      "cognitiveLevel": "understand",
      "difficultyHint": "medium",
      "questionTypeSuggestions": ["short_answer", "judgment"]
    }
  ]
}
```

### 9.5 整篇笔记自动分析

流程：

```text
用户点击“分析本篇笔记”
→ 按标题结构划分分析单元
→ 创建 AIJob
→ 分批提取重要区域和知识内容候选
→ 跨章节去重
→ 生成考点候选
→ 进入批量审核界面
```

第一版不依赖向量数据库。

分析上下文优先使用：

- 当前章节内容。
- 标题路径。
- 相邻章节摘要。
- 笔记元数据。

禁止为了整篇分析，直接将任意长度的完整笔记一次性塞给模型。

### 9.6 AI 输出校验

每个 AI 用例必须包含：

- JSON Schema 或等价结构校验。
- 枚举校验。
- 长度限制。
- 空字段检查。
- 来源 ID 合法性检查。
- 重复候选检查。
- 错误转换与重试策略。
- Prompt 版本记录。

结构校验失败时，不得写入正式数据。

- AI 输出 schema 校验。
- AIJob 状态流转。
- AI 调用失败不污染正式数据。
- 幂等请求不重复创建候选。
- 批量分析部分失败时的结果与错误记录。

### 11.2 前端

- 从标注发起提炼任务。
- Job 加载、成功、失败状态。
- 候选卡片确认、编辑、拒绝。
- 一个标注关联多个知识内容。
- 一个知识内容显示多个来源。
- 点击来源定位正文。
- 考点创建、编辑和排序。
- 候选数据默认不进入正式筛选结果。
- 删除标注不会误删知识内容。
- 删除知识内容不会误删原文标注。

### 11.3 阶段 2 验收标准

- 用户可以从任意标注生成知识内容候选。
- 用户可以把多个标注合并到同一个知识内容。
- 一个知识内容可以产生多个考点。
- AI 结果必须经过用户确认。
- 整篇笔记可以生成一批候选知识内容和考点。
- 候选与正式数据在 UI、API 和检索范围中明确区分。
- 已为未来题目与考点的多对多关系预留接口。
- 尚未要求完成完整试卷系统。

```text

重要程度
```

## 13. 阶段 3：RAG 检索与跨笔记增强

阶段 3 解决：

> 当笔记数量增多后，AI 能够从正确范围内找到相关原文、知识内容和考点，并提供可追溯引用。

### 13.1 阶段 3A：结构化切片

切片优先遵循 Markdown 结构：

```text
标题层级
段落
列表
引用
代码块
表格
公式
```

策略：

1. 先按标题章节分组。
2. 章节过长时再按块边界切分。
3. 不在代码块、表格和公式中间随意切断。
4. 每个切片保存完整标题路径。
5. 必要时保留少量重叠上下文。
6. 使用 `chunkStrategyVersion` 标记策略版本。
7. 使用 `contentHash` 支持增量更新。

禁止只按照固定字符数粗暴切分全部内容。

### 13.2 阶段 3B：关键词检索

先在现有存储能力上完成：

- 笔记标题搜索。
- 正文关键词搜索。
- 标题路径搜索。
- KnowledgeItem 搜索。
- AssessmentPoint 搜索。
- 标签与空间过滤。
- 资源类型过滤。

关键词检索是混合检索的基础，不能因为引入向量检索而废弃。

### 13.3 阶段 3C：向量检索

生产环境推荐：

```text
PostgreSQL + pgvector
```

但当前项目仍可能使用本地 JSON 存储，因此必须通过 repository interface 隔离实现。

禁止：

- 直接把大量向量数组写入现有 `knowledge-base.json`。
- 在业务 service 中写死 pgvector SQL。
- 为了 RAG 一次性重写全部知识库持久化。

推荐接口：

```text
EmbeddingProvider
EmbeddingRepository
VectorSearchRepository
```

允许实现：

```text
Keyword-only adapter        // 阶段 3A/3B
PgVector adapter            // 正式向量实现
Test in-memory adapter      // 测试
```

### 13.4 索引对象

第一批只索引：

```text
DocumentChunk
Confirmed KnowledgeItem
Confirmed AssessmentPoint
```

默认不索引：

- candidate。
- archived。
- deleted。
- stale 且无法确认来源的内容。
- 纯 UI 状态。

### 13.5 增量索引

笔记保存后：

```text
计算 noteContentHash
→ 重新生成该笔记切片
→ 比较 chunk contentHash
→ 保留未变化切片
→ 新增或更新变化切片
→ 删除失效切片
→ 只为变化资源重新生成 embedding
```

知识内容或考点更新后同理。

软删除资源必须立即从检索结果中过滤，物理删除时再清理向量记录。

### 13.6 混合检索

推荐流程：

```text
用户查询
→ 查询标准化
→ 结构化过滤
→ 关键词召回
→ 向量召回
→ 合并去重
→ 可选重排序
→ 返回引用
```

过滤条件可以包括：

```text
spaceId
noteId
folderId
tagIds
resourceType
reviewStatus
knowledgeType
timeRange
```

### 13.7 引用模型

每个检索结果至少返回：

```text
resourceType
resourceId
noteId
headingPath
quoteText
score
matchedBy           // keyword / vector / hybrid
location
```

AI 问答必须返回可点击引用。

引用点击后：

- 原文切片跳转到对应笔记位置。
- KnowledgeItem 跳转到知识内容卡片。
- AssessmentPoint 跳转到考点卡片。

### 13.8 RAG 问答

流程：

```text
用户问题
→ 混合检索
→ 选择上下文
→ AI 基于上下文回答
→ 区分“来自笔记”与“AI 补充”
→ 返回引用
```

回答规则：

- 无足够依据时明确说明资料不足。
- 不得伪造笔记引用。
- 引用必须来自实际检索结果。
- 用户可选择“仅根据我的笔记回答”或“允许 AI 补充”。

### 13.9 RAG 对知识链路的增强

完成基础问答后，再按顺序加入：

1. 新知识内容创建前的相似项查重。
2. 为知识内容补充其他笔记来源。
3. 查找前置知识、相关知识和易混知识。
4. 跨笔记生成知识内容候选。
5. 跨笔记生成考点。
6. 按主题或标签生成出题范围。
7. 为题目答案提供来源校验。
8. 错题后推荐复习原文。

## 15. 阶段 3 测试

### 15.1 切片测试

- 标题路径正确。
- 段落、列表、代码块、表格不被错误切断。
- 长章节能稳定二次切片。
- 相同正文生成相同 contentHash。
- 修改一段只影响对应切片。
- 删除章节会删除对应切片。

### 15.2 检索测试

建立固定小型测试语料，至少覆盖：

- 精确缩写：GBN、SR、TCP Reno。
- 公式与符号：`2^k`、`O(E log V)`。
- 语义查询：Prim 和 Dijkstra 的区别。
- 标签过滤。
- 笔记范围过滤。
- 已删除资源排除。
- candidate 资源默认排除。
- 关键词与向量结果合并去重。

### 15.3 引用测试

- 每个回答引用都能追溯到真实资源。
- 点击引用能跳转。
- 笔记修改后引用失效会明确提示。
- 无检索结果时不生成虚假引用。

### 15.4 索引生命周期测试

- 新建笔记后可索引。
- 修改笔记后增量更新。
- 软删除后不可检索。
- 恢复后重新可检索。
- 更换 embedding 模型后不会混用旧向量。
- 重试不会产生重复向量记录。

### 15.5 阶段 3 验收标准

- 用户可以对全知识库提问并看到引用。
- 精确关键词、缩写和公式可被可靠检索。
- 语义问题可以通过向量召回相关内容。
- 检索支持笔记、标签、资源类型等过滤。
- 笔记修改后只更新变化内容。
- 删除内容不会继续出现在结果中。
- RAG 可以辅助知识查重与跨笔记来源补充。
- 向量持久化与业务逻辑解耦。

## 16. 数据一致性与生命周期

### 16.1 删除关系

必须明确区分：

```text
删除标注
删除知识内容
删除考点
删除笔记
```

规则建议：

- 删除标注：移除正文 marker，同时删除或失效其 KnowledgeEvidence；不自动删除知识内容。
- 删除知识内容：归档其考点和证据关系；不删除原文标注。
- 删除考点：不影响知识内容。
- 删除笔记：标注和原文切片软删除或失效；由这些标注支持的知识内容进入“来源不足”状态，不应静默删除。
- 恢复笔记：重新检查标注定位并恢复索引。

### 16.2 来源不足

知识内容的全部来源失效后：

```text
sourceStatus = missing
```

前端提示用户：

- 补充来源。
- 重新绑定原文。
- 保留为用户自建知识。
- 归档。

### 16.3 幂等性

以下操作必须幂等：

- 旧知识点数据迁移。
- 笔记切片重建。
- embedding 生成。
- AI 批量提炼请求。
- 用户重复点击确认。
- Job 重试。

推荐使用：

```text
业务唯一键
contentHash
idempotencyKey
版本字段
```

## 20. 代码与文件拆分要求

### 20.1 后端

- route/controller 不写业务判断。
- application service 不依赖 HTTP 对象。
- domain 不依赖存储、AI SDK 或环境变量。
- repository 负责持久化。
- AI provider 与 embedding provider 使用 adapter。
- 复杂流程拆为步骤函数。
- 禁止循环查库或循环调用外部 API。
- 新实体按 domain/application/infrastructure/http/test 补齐。

### 20.2 前端

- API 请求进入 `src/services`。
- DOM 与业务流程编排进入 `src/controllers`。
- 纯状态、纯规则和 HTML renderer 进入 `lib`。
- 事件 binder 只转发事件，不写 API 或复杂业务。
- 不把新逻辑回填到 `client.js`。
- 新增固定视觉规则进入 styles 与 tokens。
- `knowledge-api.js` 已接近边界，新增功能前优先拆分。

### 20.3 行数

- 普通业务源文件超过 250 行必须自查职责。
- 超过 400～500 行且职责混杂时优先拆分。
- 不为了压行数机械拆文件，拆分依据必须是职责边界。
- 聚合入口可以略长，但不得混入业务逻辑。

- 不先做完整自适应掌握度算法。
- 不先做跨全库自动生成大型试卷。
- 不先做独立向量数据库集群。
- 不先做微服务拆分。
- 不让 AI 自动修改正式知识。
- 不为了 RAG 重写整个现有项目。
- 不修补或延续旧 KnowledgePoint 聚合。
- 不为没有真实数据的旧知识点 API 建兼容层。
- 不允许新 Annotation 继续依赖 `knowledgePoints / allKnowledgePoints` 双集合。
- 不把知识内容、考点、掌握度重新塞回标签体系。
- 不把 DocumentChunk 当成知识内容。
- 不把正文标注重新叫作考点或知识点。

## 25. 最终产品流程

### 手动模式

```text
用户阅读笔记
→ 标注重要内容
→ AI 提炼知识内容候选
→ 用户确认
→ AI 生成考点候选
→ 用户确认
→ 选择考点生成试卷
```

### 自动模式

```text
用户选择一篇笔记或一个范围
→ AI 分析原文
→ 生成标注、知识内容和考点候选
→ 用户批量审核
→ 形成可出题范围
```

### RAG 增强模式

```text
用户提问或选择主题
→ 混合检索相关原文、知识内容和考点
→ AI 回答或生成候选
→ 展示真实引用
→ 用户确认后进入正式知识链路
```

---

## 26. 最终架构原则

整条知识链路必须始终保持以下职责：

```text
笔记负责承载原始资料
标注负责指出重要位置
知识内容负责表达知识本身
考点负责定义学习目标
RAG 负责找到相关材料
试卷负责组织评测
答题与批改负责产生学习证据
掌握度负责反映学习结果
```

任何后续功能设计都不得重新混淆这些对象。

闭包为什么看起来总像“记住了过去”

更新时间 2026-07-02 · JavaScript / 运行机制 / 核心概念

闭包让人困惑，通常不是因为定义难，而是因为它带来一种很反直觉的感受： 一个函数明明离开了原来的位置，却还像握着过去留下来的材料。

更准确的说法是，闭包并不是“带走了一个旧值”，而是 持续保留了对那一层词法环境的访问权。 所以它读取到的，常常不是过去的快照，而是那层环境在当前时刻的状态。

Refining Note

1. 把闭包理解成“访问通道”，而不是“值的备份”

如果把闭包想成一个通道，很多现象就会自然很多。它不是偷偷复制了一份变量， 而是维持了“未来还能回来取材料”的能力。

函数被带走的，不是一张旧照片，而是一把仍然有效的钥匙。

2. 为什么 React、异步回调和事件处理总绕不开它

只要函数不是立刻执行，而是留到未来某个时刻再运行，它几乎就一定和闭包有关。 定时器、点击回调、Promise、Hook，都是在“把函数放进未来”。

stale closure 的问题，本质也不是闭包坏了，而是我们误判了它到底引用的是哪一次渲染、哪一层环境。

Applied Insight

3. 更适合记忆的一句话

闭包不是记住一个值，而是保留访问一段环境的方式。 一旦把这句话记牢，很多相关问题都会从“神秘”变成“可推演”。
